## 1. Safari Autofill Behavior with Synthetic Events

### Finding: Synthetic events DO NOT trigger Safari's native autofill UI

**Confidence: HIGH (validated)**

Safari's security model draws a hard line between "trusted user gestures" (real input events originating from physical user actions) and synthetic JavaScript events (programmatic `element.focus()`, `element.click()`, `element.dispatchEvent()`). This distinction is enforced by WebKit at the engine level.

**Key evidence:**

- The `Event.isTrusted` property in WebKit is `true` only for events generated by the user agent via user actions. Events dispatched via `EventTarget.dispatchEvent()` have `isTrusted = false`. Safari uses this flag to gate access to security-sensitive features like autofill and passkey UI. ([MDN: Event.isTrusted](https://developer.mozilla.org/en-US/docs/Web/API/Event/isTrusted))

- Safari's password/passkey autofill sheet is presented by macOS's AuthenticationServices framework, invoked only in response to trusted user activation forwarded by WebKit. Synthetic events are explicitly blocked from triggering this pathway.

- [WebKit Bug 172567](https://bugs.webkit.org/show_bug.cgi?id=172567) documents that Safari autofill does not work in form inputs that are programmatically focused or revealed, confirming this is known/intentional behavior, not a bug.

- [React issue #2125](https://github.com/facebook/react/issues/2125) (2014, still relevant) documents that Safari's autofill does not trigger `onChange` events, confirming that Safari treats autofill as a privileged native action separate from the JS event model.

- A Medium article on "tricking Safari's autofill" confirms that various workarounds (delayed focus, synthetic clicks) do not reliably trigger the native autofill UI. ([Source](https://laramo.medium.com/how-i-tricked-safaris-autofill-and-what-i-learned-doing-it-1b374153c41a))

**Safari vs Chrome comparison:**

Both browsers treat synthetic JS events as untrusted and will not display their native password manager UI in response. The key difference is in passkey upgrade flows: Safari 18 ties passkey upgrade (`navigator.credentials.create({mediation: 'conditional'})`) to password autofill events from the Passwords app, while Chrome allows passkey upgrades within ~5 minutes of any successful password sign-in.

**Implication for Safari Pilot:**

The "natural autofill appears when agent clicks a login field" flow **will not work**. The assessment's biggest risk is confirmed. Safari Pilot must use one of:

1. **Direct form fill via JS** -- bypass autofill entirely, read passwords from Keychain (see section 2), and fill form fields with `element.value = password` directly.
2. **OS-level input synthesis** -- if Safari Pilot's Swift daemon synthesizes mouse/keyboard events at the macOS level (via CGEvent or similar), these may be treated as trusted. However, this is fragile and Apple does not guarantee this behavior.
3. **User-assisted auth** -- pause, notify user, wait for them to interact with the autofill UI.

### Recommendation

Do not rely on synthetic JavaScript events to trigger autofill. Plan A should be direct Keychain read + JS form fill. Plan B should be assisted auth (user present). OS-level event synthesis is a possible Plan C but requires empirical validation.

## 2. macOS Keychain Programmatic Access

### Finding: Persistent silent access IS possible for a code-signed LaunchAgent via ACL configuration

**Confidence: MEDIUM-HIGH (validated with caveats)**

This is the most nuanced topic. The deep research initially stated that persistent access "cannot be achieved," but supplementary research reveals a more complex picture with important distinctions.

### CLI vs Security Framework

The `security find-internet-password` CLI tool is a wrapper around the Security framework. Both trigger the same system-level authorization prompts when an item's access control requires it. However:

| Aspect | `security` CLI | Security Framework (Swift) |
|--------|---------------|---------------------------|
| Prompt control | None -- shows system dialog | Can use `LAContext` with `kSecUseAuthenticationContext` to control prompt behavior |
| Error handling | Simple exit codes | Detailed `OSStatus` error codes |
| ACL management | Limited (`-T` flag on import) | Full `SecAccess`/`SecACL` APIs |
| Code signing | Runs as generic `security` binary | Can be signed with team ID matching ACL |
| Grace period | No reuse support | Supports `touchIDAuthenticationAllowableReuseDuration` |

**Recommendation:** Use the Security framework in the Swift daemon, not the CLI.

### ACL Configuration for Persistent Access

The Keychain ACL model works as follows:

1. Each keychain item has an **Access Object** (`SecAccess`) containing multiple **ACL entries** (`SecACL`).
2. Each ACL entry specifies: **authorization operations** (decrypt, export, etc.), **trusted applications** (list of code-signed app paths), and a **partition ID** (teamid, apple, or cdhash).
3. When an app not on the trusted list tries to access the item, macOS shows a prompt: "Deny / Allow / Always Allow". Choosing **Always Allow** adds the app to the ACL permanently.

**Critical findings on persistent access:**

- A **code-signed LaunchAgent** can be added to a keychain item's "Always Allow" list, after which it can access that item without prompts. This is verified by multiple StackOverflow answers and the Jamf Nation community. ([SO: How does macOS Keychain ACL determine which apps have access](https://stackoverflow.com/questions/58290058))

- The ACL entry is tied to the **binary's code signature**. If the binary is re-signed or its path changes, the ACL entry is invalidated and the prompt reappears. This means Safari Pilot daemon updates require re-approval.

- For items created by Safari Pilot's own daemon, you can set "Allow all applications" at creation time using `SecAccessCreate()` + `SecACLCreateWithSimpleContents()` with a `nil` trusted application list. ([SO: How to allow all applications to access keychain item without prompt](https://stackoverflow.com/questions/41832893))

- However, **Safari's stored website passwords** (in iCloud Keychain / Passwords app) use the **Data Protection keychain**, not the file-based keychain. ACLs are **not available** for iCloud Keychain items on macOS. The Apple docs explicitly state: "ACLs are not available in iOS or in macOS apps that use the iCloud keychain." ([Apple: Access Control Lists](https://developer.apple.com/documentation/security/access-control-lists))

**This is the critical constraint:** Safari Pilot cannot silently read Safari's stored website passwords from iCloud Keychain because ACL-based "Always Allow" doesn't work for those items. The passwords stored in the Passwords app (macOS 15+) are in iCloud Keychain.

### LaunchAgent vs LaunchDaemon for Keychain Access

- **LaunchAgent** (per-user, runs in user's GUI session): Has natural access to the user's login keychain. No special configuration needed beyond being in `~/Library/LaunchAgents/`. Do NOT set `SessionCreate: true` in the plist -- this creates a new security session that actually *prevents* keychain access. ([SO: Launchd not able to access Mac OS keychains](https://stackoverflow.com/questions/47582989))

- **LaunchDaemon** (system-wide, runs as root): Cannot access user keychains without unlocking them, which requires the user's password. Not suitable for Safari Pilot's use case.

**Safari Pilot's daemon already runs as a LaunchAgent**, so it has the right session context.

### macOS Passwords App (macOS 15 Sequoia) Impact

The new Passwords app is a UI change, not an API change. It provides a centralized interface for managing passwords, passkeys, and credentials stored in iCloud Keychain. The underlying programmatic access model remains unchanged -- `SecItemCopyMatching` and related APIs still work the same way. No new entitlements or APIs were introduced that would allow background processes to bypass consent requirements. ([Source: Eclectic Light](https://eclecticlight.co/2024/09/26/inside-sequioas-new-passwords-app/))

### Practical Architecture for Safari Pilot

Given that iCloud Keychain items cannot have ACLs configured for silent access:

1. **Safari Pilot's own credential store**: Create keychain items owned by the daemon with "Allow all" ACLs. Users can import/store credentials specifically for Safari Pilot automation. These CAN be accessed silently.

2. **Reading Safari's stored passwords**: Will ALWAYS require a user prompt (Touch ID or password). This prompt comes from the system, cannot be suppressed, and appears each time unless `touchIDAuthenticationAllowableReuseDuration` is used (see section 8).

3. **The `security find-internet-password` command**: Returns passwords from the **login keychain** (file-based), NOT from iCloud Keychain. Website passwords stored by Safari in the Passwords app are in iCloud Keychain and may not be accessible via this command at all.

### Recommendation

Implement a **dedicated credential store** within Safari Pilot's own keychain namespace. The daemon creates items with permissive ACLs at creation time, enabling silent reads. Provide a UI/CLI for users to import credentials into this store. For accessing Safari's existing stored passwords, accept that a Touch ID prompt will appear and use `touchIDAuthenticationAllowableReuseDuration` (max 5 minutes) to reduce prompt frequency during a session.

## 3. Passkey/WebAuthn on Safari

### Finding: Per-interaction biometric verification is enforced, with one important edge case

**Confidence: HIGH (validated)**

### userVerification Behavior on Safari/macOS

Safari's platform authenticator (macOS Secure Enclave) enforces user verification for every passkey operation. However, the behavior varies based on the `userVerification` parameter and device configuration:

| `userVerification` | Touch ID Available | Touch ID Unavailable (clamshell) |
|---|---|---|
| `required` | Touch ID prompt, UV=1 | **Safari bug: completes with UV=0** (violates spec) |
| `preferred` (default) | Touch ID prompt, UV=1 | System password prompt, UV=1 |
| `discouraged` | **May still prompt** depending on authenticator | User presence only, UV=0 |

**Critical Safari-specific behavior:** When `userVerification = "required"` and Touch ID is not available (clamshell mode, external display, Mac Studio), Safari on macOS completes the ceremony with `UV=0` instead of failing as the spec requires. Chrome correctly prompts for the system password in the same scenario. This is a known spec violation documented by Corbado with test evidence. ([Source: Corbado WebAuthn User Verification](https://www.corbado.com/blog/webauthn-user-verification))

**`userVerification = "discouraged"` on Safari:** The relying party requests no verification, but Safari's platform authenticator may still perform it. The spec says "discouraged" is a hint, not a requirement. On macOS with Touch ID available, Safari typically shows a minimal Touch ID prompt (user presence confirmation) even with `discouraged`.

### Conditional Mediation (Passkey Discovery)

Safari 18 on macOS 15 fully supports conditional mediation (`mediation: 'conditional'` with empty/omitted `allowCredentials`). This enables:

1. Available passkeys appear as suggestions in the AutoFill UI, attached to username fields
2. Checked via `PublicKeyCredential.isConditionalMediationAvailable()`
3. Still requires user interaction with the system-provided UI for final selection

This is relevant for Safari Pilot because conditional mediation means passkeys can appear in the same UI as passwords. The agent cannot interact with this UI programmatically, but can detect the outcome.

### Passkey Sheet: System UI, Not DOM

The passkey selection sheet is rendered by macOS's AuthenticationServices framework, NOT part of the web page DOM. It is:

- Not accessible via JavaScript
- Not scriptable via AppleScript
- Not accessible via standard Accessibility APIs (it's a secure system UI element)
- Immune to automation -- this is a deliberate security measure

### Apple Developer Forum Thread (Jan 2026)

A [developer forum thread](https://developer.apple.com/forums/thread/813784) reports that `userVerification` is ignored during Passkey Autofill in non-Safari browsers on macOS. This confirms that iCloud Keychain's passkey handling has known inconsistencies across browsers.

### Implication for Safari Pilot

Fully automated passkey authentication is confirmed impossible. The assessment's initial finding is validated. Safari Pilot must:

1. **Detect** when a site requests WebAuthn (can monitor for `navigator.credentials.get()` calls via the Web Extension)
2. **Pause and notify** the user that biometric interaction is required
3. **Wait for completion** by observing post-authentication side effects (URL change, network requests, DOM changes)
4. **Resume** automation after detecting successful auth

### Recommendation

Implement passkey flows as part of the "Assisted Auth" pillar. The agent detects WebAuthn invocation, pauses, notifies the user, and monitors for completion signals. No attempt should be made to interact with the passkey sheet.

## 4. OAuth Popup Windows in Safari + AppleScript

### Finding: AppleScript CAN enumerate and monitor OAuth popups, with important caveats

**Confidence: HIGH (validated)**

### AppleScript Capabilities

AppleScript can reliably:

1. **Enumerate all Safari windows**: `tell application "Safari" to every window` -- includes popup windows opened by `window.open()`
2. **Read popup URLs**: `URL of document of window` -- enables monitoring for OAuth callback redirects
3. **Execute JavaScript in popups**: `do JavaScript` in a specific tab/window -- subject to same-origin policy
4. **Close popups**: After capturing the authorization code

This makes the following flow viable:
```
1. Agent clicks "Sign in with Google" on the main page
2. Safari opens OAuth popup to accounts.google.com
3. AppleScript polls `every window` to detect the new popup
4. AppleScript monitors the popup's URL for the callback redirect
5. When URL matches callback pattern (e.g., contains `code=`), extract the auth code
6. Close the popup, resume main page automation
```

### Safari Popup Blocker Interaction

Safari blocks `window.open()` calls that lack a **trusted user gesture** origin. This means:

- Popups triggered by a real user click: **Allowed**
- Popups triggered synchronously from a click handler: **Allowed**
- Popups triggered by `setTimeout` or async code after a click: **May be blocked**
- Popups triggered with no user gesture (e.g., on page load): **Blocked** (shows Block/Allow dialog)

**For Safari Pilot:** If the agent's `safari_click` uses JS `element.click()` (synthetic), the resulting popup may be blocked because the click is not a trusted gesture. If it uses AppleScript `click at {x,y}` or CGEvent-level clicking, it may be treated as trusted.

The popup blocker presents a "Block / Allow" dialog. If it appears, AppleScript/Accessibility APIs could potentially dismiss it, but this adds fragility.

### OAuth Provider Flow Patterns

| Provider | Default Mode | Popup Support | Redirect Support | Force Redirect |
|----------|-------------|---------------|------------------|----------------|
| Google | Popup (default for web) | Yes | Yes | `redirect_uri` param or `ux_mode: 'redirect'` in JS SDK |
| GitHub | Redirect | App wraps in popup | Yes (native) | Always redirect by default |
| Microsoft (MSAL) | Popup (default) | Yes | Yes | `loginRedirect()` instead of `loginPopup()` |
| Apple | Popup (JS SDK) | Yes | Yes | Configurable in JS SDK |

**Key insight:** All major providers support redirect mode. If Safari Pilot can force redirect mode (by modifying the OAuth initiation URL or intercepting the SDK configuration), popups can be avoided entirely.

### Recommendation

Implement a two-pronged approach:

1. **Popup adoption** (primary): Extend `TabOwnership` to track parent-child window relationships. When an agent-owned tab spawns a popup, auto-adopt it. Use AppleScript polling to monitor the popup URL for callback completion.

2. **Redirect mode forcing** (optimization): Where possible, modify OAuth initiation to use redirect mode instead of popup mode. This keeps the flow within a single tab and avoids popup blocker issues entirely.

## 5. Sign in with Apple on Safari

### Finding: Presentation mode varies; system sheet is NOT scriptable, web-based flow is partially automatable

**Confidence: MEDIUM-HIGH (validated)**

### UI Presentation Modes

"Sign in with Apple" on macOS Safari can appear in two forms:

1. **Native system sheet** (via `ASAuthorizationController`): When a site uses Apple's native AuthenticationServices APIs, Safari presents a native macOS sheet. This sheet can include Sign in with Apple alongside passkey and password options. It is rendered by the OS, not by Safari's web renderer.

2. **Web-based popup**: When a site uses Apple's JavaScript SDK for web, the flow opens a popup window to `appleid.apple.com`. This is a standard web page.

The native system sheet is the more common case on Safari, as Safari optimizes for the native flow.

### Automation Feasibility

| Aspect | Native Sheet | Web Popup |
|--------|-------------|-----------|
| AppleScript control | **No** | URL monitoring only (same-origin) |
| Accessibility APIs | **Limited/No** -- security-hardened | Potentially via page elements |
| JavaScript interaction | **No** | Subject to same-origin policy |
| Completion detection | Monitor for post-auth URL/network | Monitor for callback redirect |

### Authenticated Session Behavior

When the user is already signed into their Apple ID on the Mac:

1. The sheet appears but **pre-populates** the Apple ID email
2. The user still must **confirm** by clicking "Continue"
3. A **Touch ID or password prompt** typically appears for the cryptographic signature
4. There is NO fully automatic "just passes through" mode for Sign in with Apple

**Apple's documentation is explicit:** The `ASAuthorizationController` presents a modal sheet that requires user interaction via `presentationAnchor(for:)`. ([Apple: Implementing User Authentication with Sign in with Apple](https://developer.apple.com/documentation/AuthenticationServices/implementing-user-authentication-with-sign-in-with-apple))

### Implication for Safari Pilot

Sign in with Apple flows should be treated as "assisted auth" -- the agent detects the Sign in with Apple button/flow, pauses, notifies the user, and waits for completion. The assessment's initial finding is confirmed.

### Recommendation

Treat Sign in with Apple as a special case of assisted auth. The agent:
1. Detects the Sign in with Apple button (can identify by standard CSS classes `signin-button` or button text)
2. Pauses and notifies: "Sign in with Apple requires your interaction"
3. Waits for completion by monitoring for the post-auth redirect or token exchange
4. Resumes automation

## 6. Auth Completion Detection Strategies

### Finding: Multi-signal approach with network interception as primary is most reliable

**Confidence: HIGH (validated)**

### Signal Comparison

| Signal | Reliability (Traditional) | Reliability (SPA) | Implementation Complexity |
|--------|--------------------------|--------------------|----|
| URL change | **HIGH** -- definitive for redirect flows | **LOW** -- SPAs often don't navigate | Low |
| Network request (token exchange) | **HIGH** | **HIGH** -- best signal for SPAs | Medium |
| DOM marker (e.g., "Logout" button) | **MEDIUM** -- can have race conditions | **MEDIUM-HIGH** | Medium |
| Cookie/localStorage mutation | **MEDIUM** -- race conditions possible | **MEDIUM** -- but token storage varies | Low |
| Page title change | **LOW** -- unreliable, not universal | **LOW** | Low |

### Playwright's Best Practices (Authoritative)

Playwright's documentation provides the most mature patterns:

1. **`storageState`**: Playwright saves and reloads authentication state (cookies + localStorage) to avoid re-authenticating. Safari Pilot can implement an equivalent by extracting and restoring cookies/storage. ([Playwright: Authentication](https://playwright.dev/docs/auth))

2. **Race-free waiting pattern**: Always initiate the wait BEFORE triggering the action:
   ```javascript
   await Promise.all([
     page.waitForNavigation(),
     page.click('button[type=submit]')
   ]);
   ```

3. **`waitForResponse`**: Wait for a specific API endpoint to return:
   ```javascript
   await page.waitForResponse(resp =>
     resp.url().includes('/api/auth/token') && resp.status() === 200
   );
   ```

4. **Avoid `networkidle`**: Playwright explicitly discourages `waitForLoadState('networkidle')` as it is flaky due to background analytics, service workers, and polling connections. ([Playwright: Best Practices](https://playwright.dev/docs/best-practices))

### Safari Web Extension Network Interception

**Critical finding on webRequest API availability:**

Safari 18 **re-enabled webRequest API support for Manifest V3 extensions**. A WebKit contributor confirmed this in April 2025: "Safari 18 dropped the requirement for a persistent page to use webRequest -- allowing it to work in manifest v3 extensions like Chrome and Firefox." ([GitHub: mdn/browser-compat-data#24571](https://github.com/mdn/browser-compat-data/issues/24571))

However, there are important limitations:
- **`BlockingResponse` is NOT supported** in Safari -- extensions can observe but not modify/block requests
- **`opt_extraInfoSpec` is not supported** for any events
- **`cookies` header** is not available in `webRequest.HTTPHeaders`
- **iOS does not support webRequest** at all (macOS only)

The observing capability is sufficient for auth detection -- Safari Pilot's Web Extension can listen for `webRequest.onCompleted` events on token exchange endpoints.

### Recommended Multi-Signal Strategy for Safari Pilot

```
Priority 1: Network request interception (via Web Extension webRequest.onCompleted)
  - Monitor for POST to known token endpoints (/oauth/token, /api/auth, etc.)
  - Monitor for 302 redirects from OAuth providers back to app domain
  - Most reliable for both traditional and SPA flows

Priority 2: URL monitoring (via AppleScript polling)
  - Poll `URL of document of window` at 100-500ms intervals
  - Match against known post-login URL patterns
  - Works for redirect-based flows

Priority 3: DOM markers (via Web Extension content script)
  - Observe DOM for appearance of authenticated-state elements
  - MutationObserver watching for user avatar, logout button, dashboard content
  - Fallback for SPAs that don't change URL

Priority 4: Cookie/storage polling (via Web Extension or JS execution)
  - Poll for session cookies or localStorage tokens
  - Last resort, most prone to race conditions
```

### Recommendation

Implement the multi-signal strategy with network interception as the primary signal. The Web Extension's `webRequest.onCompleted` listener provides the most reliable cross-site detection. URL polling via AppleScript serves as a complementary signal. DOM observation and storage polling are fallbacks.

## 7. macOS Notification APIs for Swift Daemons

### Finding: UserNotifications framework works from LaunchAgent but requires careful setup

**Confidence: HIGH (validated)**

### UserNotifications Framework from LaunchAgent

The `UserNotifications` framework CAN be used from a LaunchAgent, with these requirements:

1. **Process must run in the user's GUI session** -- LaunchAgent is correct (LaunchDaemon is not)
2. **Must request authorization** via `UNUserNotificationCenter.current().requestAuthorization(options:)` -- prompts user on first run
3. **App identifier required** -- the LaunchAgent needs a bundle identifier for notification center to track it
4. **Entitlement for sandboxed processes**: If sandboxed, needs `com.apple.security.notifications` entitlement

**Apple Developer Forum thread (Oct 2025)** confirms the difficulty: a developer reports `UNUserNotificationCenter` authorization works from the main app but not from its LaunchAgent helper, even when the agent is within the app bundle. The issue appears to be that the notification center ties to the bundle ID, and a LaunchAgent without its own bundle may inherit the wrong identity. ([Apple Forums: Posting notifications from a launch agent](https://developer.apple.com/forums/tags/usernotifications?page=2))

### Action Buttons in Notifications

Yes, notifications CAN include action buttons:

1. Define `UNNotificationCategory` with associated `UNNotificationAction` objects
2. Register categories via `center.setNotificationCategories(_:)`
3. Handle responses via `UNUserNotificationCenterDelegate.userNotificationCenter(_:didReceive:withCompletionHandler:)`
4. The delegate callback runs even if the process is not foregrounded -- the system launches the LaunchAgent in background

This enables interactive notifications like: **"Safari Pilot: Authentication required for github.com"** with buttons like "Open Safari" or "Dismiss".

### `osascript` Fallback

`osascript -e 'display notification "message" with title "title"'` works as a simple fallback but:

- **No action buttons** -- purely informational
- Only works in GUI session (not from LaunchDaemon or headless context)
- Requires Automation or Accessibility permissions
- Not reliable for critical alerts

### Recommended Architecture for Safari Pilot

Safari Pilot's daemon already runs as a LaunchAgent in the user session. The recommended approach:

1. **Primary**: Use `UserNotifications` framework directly from the daemon. Give the daemon its own bundle identifier in its Info.plist.
2. **Interactive notifications**: Define a `SAFARI_PILOT_AUTH_REQUIRED` category with "Open Safari" action button.
3. **Fallback**: If `UserNotifications` authorization is denied, fall back to `osascript` for basic notifications.
4. **Also consider**: Sending the notification through Claude Code's own mechanisms (console output, human approval flow) since Safari Pilot operates within Claude Code.

### Recommendation

Implement UserNotifications in the Swift daemon with action buttons. Also provide notification state through the MCP tool responses so Claude Code can display auth-required messages in its own interface.

## 8. Touch ID Grace Periods

### Finding: LAContext provides a configurable grace period (up to 5 minutes) that applies to device unlock reuse, with specific constraints

**Confidence: HIGH (validated with Apple documentation)**

### `touchIDAuthenticationAllowableReuseDuration`

Apple provides an official, documented grace period mechanism:

```swift
var touchIDAuthenticationAllowableReuseDuration: TimeInterval { get set }
```

**Official Apple documentation states:**

> "If the user unlocks the device using Touch ID within the specified time interval, then authentication for the receiver succeeds automatically, without prompting the user for Touch ID. This bypasses a scenario where the user unlocks the device and then is almost immediately prompted for another fingerprint."

**Key parameters:**
- **Default value**: `0` (no reuse -- every access prompts)
- **Maximum value**: `LATouchIDAuthenticationMaximumAllowableReuseDuration` = **5 minutes (300 seconds)**
- **Available since**: iOS 9.0, macOS 10.12
- **Applies to**: Biometric (Touch ID/Face ID) authentication reuse

([Apple: touchIDAuthenticationAllowableReuseDuration](https://developer.apple.com/documentation/localauthentication/lacontext/touchidauthenticationallowablereuseduration))
([Apple: Accessing Keychain Items with Face ID or Touch ID](https://developer.apple.com/documentation/LocalAuthentication/accessing-keychain-items-with-face-id-or-touch-id))

### Scope and Limitations

**What it does:**
- Reuses a recent device unlock (Touch ID/Face ID) to skip a subsequent biometric prompt
- Applies specifically to `LAContext.evaluatePolicy()` calls
- Can be passed to `SecItemCopyMatching` via `kSecUseAuthenticationContext`

**What it does NOT do:**
- Does NOT apply to arbitrary keychain reads -- only to items accessed through an `LAContext` with this property set
- Does NOT create a system-wide grace period -- scoped to the specific `LAContext` instance
- Does NOT reuse a previous keychain authentication -- only reuses device unlock events
- Does NOT work reliably with `SecItemCopyMatching` in all scenarios -- multiple StackOverflow reports of `errSecAuthFailed` when the grace period should apply ([SO: Get keychain item with LAContext and touchIDAuthenticationAllowableReuseDuration](https://stackoverflow.com/questions/37949518))

**Apple's own documentation clarifies:**

> "Note that this grace period applies specifically to device unlock with Touch ID, not keychain retrieval authentications."

This is a crucial distinction. The grace period reuses the **device unlock** biometric, not a previous keychain access biometric.

### Known Issues

- A 2016 StackOverflow thread reports that setting `touchIDAuthenticationAllowableReuseDuration` and then calling `SecItemCopyMatching` with the same `LAContext` returns `errSecAuthFailed` when the grace period should apply. Multiple developers confirmed this issue. As of 2019, a Square/Valet GitHub issue suggests it may only work reliably on Touch ID devices, not Face ID devices.

- iOS 13.2+ documentation updated the language to "unlocked by biometry" (not just Touch ID), suggesting broader support was intended.

### Apple Watch Unlock Interaction

Apple Watch Auto Unlock:
- **CAN** unlock the Mac (bypassing login password)
- **CAN** approve some system-level prompts
- **CANNOT** substitute for Touch ID/Face ID when accessing biometry-protected keychain items
- **CANNOT** be used to approve Keychain reads for protected data

([Apple: Unlock your Mac and approve requests with Apple Watch](https://support.apple.com/en-md/guide/mac-help/mchl4f800a42/mac))

### Implication for Safari Pilot

The grace period is useful but not as powerful as hoped:

1. If the user recently unlocked their Mac with Touch ID, Safari Pilot's daemon can set `touchIDAuthenticationAllowableReuseDuration = 300` (5 minutes) on its `LAContext` to potentially skip the biometric prompt for keychain reads.
2. This only helps for items accessed through that specific `LAContext` -- not for general Keychain reads.
3. There are known bugs where this doesn't work reliably with `SecItemCopyMatching`.
4. The 5-minute window is useful for batch operations (read multiple credentials quickly after one unlock).

### Recommendation

Implement `touchIDAuthenticationAllowableReuseDuration = LATouchIDAuthenticationMaximumAllowableReuseDuration` on the daemon's `LAContext` to maximize the reuse window. But design the UX assuming prompts WILL appear -- the grace period is an optimization, not a guarantee. Test empirically on macOS 15+ as reliability has varied across OS versions.

## 9. Integrated Recommendations

### Architecture Decision: "Observe, Don't Interact"

The overarching finding across all 8 research topics is that Apple's security model makes direct interaction with authentication UI elements impossible for automation. The winning strategy is to **observe the side effects of successful authentication** rather than trying to drive the auth process itself.

### Recommended Implementation Priority

#### Priority 1: Auth Completion Detection (foundation for everything else)

Implement the multi-signal detection system:
- Web Extension `webRequest.onCompleted` listener (Safari 18+ supports this in MV3)
- AppleScript URL polling for redirect-based flows
- DOM MutationObserver via content script for SPA post-auth state changes

This is the prerequisite for all other auth features.

#### Priority 2: Assisted Auth Mode (pause/notify/wait/resume)

The "human in the loop" flow for passkeys, Sign in with Apple, MFA, and any auth the agent cannot automate:
- Detect auth wall (login form, passkey prompt, OAuth redirect)
- Notify via `UserNotifications` framework (with action buttons) AND via MCP tool response
- Wait for completion using the multi-signal detection system
- Resume automation

#### Priority 3: Popup Adoption for OAuth

Extend `TabOwnership` to auto-adopt popups from agent-owned tabs:
- AppleScript `every window` polling to detect new popups
- URL monitoring for callback redirect
- Auth code extraction from redirect URL
- Where possible, force redirect mode to avoid popups entirely

#### Priority 4: Keychain Read Mode (opt-in, for passwords only)

The most complex and risk-laden feature:
- Create Safari Pilot's own keychain namespace with permissive ACLs
- Provide CLI/UI for users to store automation credentials
- Use `LAContext` with `touchIDAuthenticationAllowableReuseDuration = 300` for grace period
- Accept that reading Safari's stored passwords (iCloud Keychain) will always require a user prompt
- Fill forms directly via JavaScript (`element.value = ...`)

### What NOT to Build

1. **OS-level event synthesis to trigger autofill** -- fragile, undocumented, may break with any macOS update
2. **Accessibility API interaction with passkey/Sign in with Apple sheets** -- these are security-hardened system UI, explicitly designed to resist automation
3. **Keychain item ACL modification for Safari's stored passwords** -- iCloud Keychain items don't support ACLs
4. **Any logic depending on Touch ID grace periods for reliability** -- use as optimization only

## 10. Updated Assumption Status

| # | Assumption | Status | Evidence |
|---|-----------|--------|----------|
| 1 | Safari shows autofill UI on synthetic JS focus/click | **INVALIDATED** | WebKit enforces trusted gesture requirement. Bug 172567, MDN isTrusted docs, autofill research all confirm. |
| 2 | `security find-internet-password` can read Safari's stored passwords | **PARTIALLY INVALIDATED** | Can read from login keychain (file-based) but Safari stores passwords in iCloud Keychain (Data Protection) which has different access rules and no ACL support. |
| 3 | Agent can see popup windows via `safari_list_tabs` | **LIKELY VALID** (needs empirical test) | AppleScript can enumerate all Safari windows including popups. If `safari_list_tabs` uses the same mechanism, popups should be visible. |
| 4 | Auth completion can be reliably detected | **VALIDATED** | Multi-signal approach (network + URL + DOM) provides reliable detection. Safari 18 re-enabled webRequest for MV3 extensions. |
| 5 | Touch ID approval has a grace period for Keychain reads | **PARTIALLY VALIDATED** | `touchIDAuthenticationAllowableReuseDuration` provides up to 5 minutes, but only reuses device unlock (not keychain auth), has known bugs with `SecItemCopyMatching`, and doesn't apply to iCloud Keychain items. |

### New Assumptions Discovered

| # | New Assumption | Status | Notes |
|---|---------------|--------|-------|
| 6 | Safari Pilot's AppleScript clicks are treated as trusted user gestures | **UNVALIDATED** | If AppleScript's `click` goes through the Accessibility framework, it may or may not satisfy WebKit's trusted gesture check. Must test empirically. |
| 7 | Safari Web Extension webRequest.onCompleted fires for OAuth redirects | **LIKELY VALID** | Safari 18 supports webRequest in MV3. Observing (not blocking) is confirmed supported. Needs empirical test with actual OAuth flow. |
| 8 | Safari Pilot's daemon can use UserNotifications without an app bundle | **UNCERTAIN** | Apple Forum thread suggests difficulties. May need to wrap daemon in a minimal app bundle or use `osascript` fallback. |
