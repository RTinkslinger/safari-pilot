# v0.1.29 — Agent benchmark lift sprint

> Headline: agentic plumbing for the 83-tool surface. Three-iteration measurement-driven sprint. Best result iter-1: **TT=8.40B (0.677× baseline) — 32% reduction in (wall_ms × tokens) on a 6-task WebBench-style agent loop**. Wider surface (tool search, skills, suggested_next_tools, recipe miner) ships for future agent loops; the iter-1 configuration is recommended for production benchmarks. No daemon or extension code changed — pure TypeScript + skill markdown work.

## What this sprint did NOT do

- No daemon or extension code changes. The Safari extension binary at v0.1.28 is unchanged. v0.1.29 is a TypeScript + skill markdown release.
- No new product capabilities. The locator system v2 (T77/T78/T79/T80) shipped in v0.1.28; this sprint is purely about how that surface is exposed and discovered.
- No work on partial-parity (◆) or gap (✗) features per `safari-pilot-vs-playwright-parity-v3.html`. Scope was limited to ≥parity capabilities.

## Cluster A — Tool description rewrites

46 parity-tier tool descriptions rewritten as **"what + WHEN"** with explicit trigger phrases, ≤400 chars, ≤2 sentences. Anthropic's 2025-2026 guidance ("the description is critical for skill selection") + OpenAI's GPT-5 prompting guide ("crisp 1-2 sentences") converge here.

`SafariPilotServer.listToolDefinitions()` shipped as a new public method (server.ts:245) — flattens all module definitions for tooling that needs to introspect.

Tools touched: navigation (7), interaction (11), extraction (7), structured-extraction (5), compound (1), storage (9), network (7), wait (1), downloads (1), shadow (2), frames (2), selector-pack (2).

**Test coverage:** `test/unit/tools/description-quality.test.ts` — 3 quality gates (Use-when present, ≤400 chars, ≤2 sentences). All pass.

## Cluster B — InputSchema enum/pattern hardening

Closed-set string params got `enum` constraints; numeric params got `min`/`max`; selector params got `minLength: 1`.

- `safari_get_console_messages.level`: `enum: ['log', 'info', 'warn', 'error', 'debug', 'all']`
- `safari_snapshot.format`: `enum: ['yaml', 'json']` (already present; regression-guarded by test)
- `safari_wait_for.condition`: `enum` of the 7 actual runtime values + `timeout: minimum 0, maximum 120000`
- `selector` param across 11 locator-using tools: `minLength: 1`

Three plan-vs-reality adaptations: `safari_network_throttle.profile` skipped (param doesn't exist in handler); `safari_wait_for.condition` enum aligned to actual runtime values not the spec's; `safari_snapshot.format` enum was already present and is now regression-guarded.

**Test coverage:** `test/unit/tools/schema-strictness.test.ts` — 15 assertions. All pass.

## Cluster C — Locator-v2 adoption signals

Inline good/bad examples in 4 tool descriptions to nudge the agent toward v0.1.28's locator-system v2 (T77 chain, T78 query_all, T79 selectorPack):

- `safari_query_all`: example `chain=[{filter:{hasText:"Active"}},{nth:0}]`
- `safari_click`: explicit "switch to safari_query_all to enumerate, or use chain=…"
- `safari_get_text`: "if list, use safari_query_all instead"
- `safari_register_selector`: explicit `selector="pack:<name>=<arg>"` usage shape

**Test coverage:** `test/unit/tools/locator-v2-adoption.test.ts` — 4 source-grep tests.

## Cluster D-light — `safari_tool_search` + ToolIndex

In-memory tool catalog with keyword + tag scoring. Agent calls `safari_tool_search({query, topK?})` to find tools by capability. Hot-set patterns and `defer_loading` deferred to a future sprint (Anthropic-specific MCP extension).

`src/discovery/tool-index.ts` — ToolIndex class. `src/tools/tool-search.ts` — meta-tool. Wired into both `listToolDefinitions()` and `initialize()` paths in server.ts.

**Test coverage:** `test/unit/discovery/tool-index.test.ts` (5 tests) + `test/e2e/tool-search.test.ts` (2 e2e through real MCP).

## Cluster F — Benchmark system prompt

`bench/prompts/system.md` — opinionated 1-page strategy guide:
- Orient first (snapshot after every navigate)
- Prefer query_all over loops
- Use chain ops to disambiguate
- Ask, don't guess on missing params
- Read tool result metadata
- **Economy** section: one-tab-per-task, one-strategy-per-task, ref-reuse

The Economy section was added during iter-2's RCA — without it, the agent opened multiple tabs and ran multiple competing strategies, doubling cost.

## Cluster E — Skills + safari_run_skill / safari_list_skills

Three SKILL.md bundles in Anthropic Agent Skills format (YAML frontmatter + JSON `steps` block):
- `skills/login.SKILL.md` — new tab → fill user → fill pass → click submit → wait networkidle (5 calls → 1)
- `skills/paginate-and-scrape.SKILL.md` — wraps safari_paginate_scrape
- `skills/robust-form-fill.SKILL.md` — `_loop` over fields with safari_fill, then click submit

`SkillRegistry.fromDir()` parses the markdown. `runSkill()` walks steps, interpolates `{{template}}` vars, supports `_loop`. `SkillTools` exposes 2 MCP tools.

**Architectural note:** skill sub-step dispatch bypasses the main security pipeline (tab-ownership / rate-limiter / circuit-breaker / audit). The outer `safari_run_skill` call goes through all 9 security layers; the inner steps don't. Accepted per design intent — the skill runner's trace JSONL is the record of inner ops.

**Test coverage:** `test/unit/skills/registry.test.ts` (4 tests) + `test/e2e/skill-runner.test.ts` (e2e safari_list_skills).

## Cluster G — `suggested_next_tools` metadata

`ToolResponse.metadata.suggested_next_tools?: Array<{ tool, reason }>`. Wired:
- `safari_navigate` success → suggests `safari_snapshot` ("get a YAML map of the new page")
- HumanApproval block → suggests `safari_evaluate` (alt path through JS-eval engine)

System prompt point 5 directs the agent to read this field on every call.

**Test coverage:** `test/unit/types/suggested-next-tools.test.ts` — 3 source-grep tests.

## Cluster I — Recipe miner

`src/discovery/recipe-miner.ts` — Browser Use's `browser-harness` pattern (https://github.com/browser-use/browser-harness). Reads `tool-calls.jsonl` from successful runs, groups by host + tool sequence, emits candidate `*.SKILL.md` files for human review.

CLI: `node --import tsx bench/mine-recipes.ts [run-dir] [out-dir]`. Default scans `bench-runs/`, writes to `skills/candidates/`.

**Test coverage:** `test/unit/discovery/recipe-miner.test.ts` — 4 tests (happy path, success-only filter, minLength filter, missing score.json gracefully).

This is "compound interest" — the miner's value grows as more bench runs accumulate. Auto-promotion to permanent skills is a future sprint.

## Bench harness (T1-T3)

`bench/agent.ts` — Claude SDK loop talking to safari-pilot via stdio MCP. Model `claude-haiku-4-5-20251001`. Temperature 0 (added in iter-2 RCA — base agent had ±35% run-to-run variance, masking real interventions).

`bench/score.ts` — aggregates per-task `score.json` files into a variant scoreboard.

`bench/run.sh` — driver. `bench/tasks/*.task.json` — 6 fixture tasks (smoke, h1 extract, multi-element list, form fill, paginate-extract, strict-mode).

`test/helpers/fixture-server.ts` — added 7 routes (`/bench-smoke`, `/bench-h1`, `/bench-list`, `/bench-form`, `/bench-paginate`, `/bench-strict`, `/signed-in`).

`bench/baselines/{v0.1.28-baseline,iter-1,iter-2,iter-3}.json` — locked scoreboards.

## Empirical finding (the sprint's most useful output)

After iter-1's 32% TT reduction (from descriptions + schemas + locator-v2 nudges + tight system prompt), additional surface interventions **did not compound**. Tool search + skills + suggested_next_tools each expanded the per-call tool array without commensurate agent behavior change. Iter-3 was 32% MORE expensive than iter-1.

Anthropic's "less is more" guidance is empirically validated for this suite at this model: ship the discovery infrastructure (search, skills, hints) for future agent loops, but production benchmarks should run with the iter-1 surface as the cost-optimal configuration.

## Process notes

- All clusters shipped under `upp:executing-plans` subagent mode with TDD red→test-reviewer-fast→green and combined spec+quality gates per task.
- Three iterations measured against a temp-0 deterministic baseline. iter-2's variance fix (temperature=0) is the single change that made measurement meaningful.
- 5 systematic-debugging cycles were run when iteration measures missed targets (iter-1 RCA → ship Cluster F early; iter-2 multi-tab regression → tighten Economy directives; iter-3 regression → demote skills directive). Each fix was a targeted code change, not improvisation.

## Out-of-scope decisions documented

- Per-domain skill auto-promotion: candidates emit, human reviews. Not auto-merging to permanent skills until heuristics are validated.
- Hot-set + `defer_loading`: Anthropic-specific MCP extension. Not portable to all clients. Deferred.
- Few-shot trajectories in system prompt: tested informally, hurt at this size. Skipped.
