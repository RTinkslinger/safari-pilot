## New: `safari_file_upload` (T41 / 5A.1)

Programmatic file upload to standard `<input type=file>` elements,
including those hidden behind `<label>` (use `force: true`).

### Limits
- 25 MiB per file (`FILE_UPLOAD_FILE_TOO_LARGE.cap = 26_214_400`)
- 4 files per call (`FILE_UPLOAD_TOO_MANY_FILES`)
- Default `validationProbeMs = 200` (set `0` to skip validation read-back)

### Supported
- Single-file upload to `<input type=file>`
- Multi-file upload (comma-separated MIMEs) to `<input type=file multiple>`
- Hidden inputs behind `<label>` via `force: true`
- React Hook Form (`useForm` + `register`) — locator path resolves the inner `<input>`
- `clear: true` — empty the input via the same DataTransfer path
- Validation read-back: `input.validationMessage` + sibling `[role=alert]` / `[aria-invalid]`
- `mimeOverrides` for ambiguous extensions

### NOT supported in v1
- Drag-and-drop dropzones (no fallback `<input>` traversal)
- Custom file pickers that bypass `<input type=file>`
- Native OS file dialogs (no AX automation)
- Locator types other than `selector` / `xpath` / `ref` inside the extension JS injection (role/text/label/placeholder require shared-locator extraction into the extension bundle — deferred)

### Known limitations exposed by smoke (open follow-ups)

- **Site-CSP can block content-isolated → daemon fetch.** Phase 0's spike validated
  the architecture against the in-repo fixture server, which has permissive CSP.
  The fixture-based e2e suite cannot exercise this gap — it is invisible in
  green tests. Real-world sites with strict CSP (Gmail confirmed) reject
  `content-isolated.js`'s `fetch('http://127.0.0.1:19475/file-bytes/<token>')`.
  Candidate fixes: move byte transport into MAIN-world via fragmented
  postMessage; add a fixture variant with strict CSP to extend e2e coverage.
  Filed as a follow-up TRACKER item.

- **Wake-without-reconcile (extension lifecycle).** Recurred during this smoke
  run despite T60's fix. Alarm fires (`lastAlarmFireTimestamp` advances) but
  `lastReconcileTimestamp` does not. Recovery requires a manual extension
  toggle in Safari Settings. Not a 5A.1 issue, but discovered during 5A.1
  smoke — filed as a follow-up TRACKER item.

### Architecture (Approach 3 — out-of-band byte fetch)

- TS handler reads bytes via Node `fs` → ships base64 to daemon via NDJSON `stage_file`
- Daemon stores in token-keyed `actor`-protected map with 60s TTL (`FileStagingStore`)
- Extension `content-isolated.js` fetches `GET http://127.0.0.1:19475/file-bytes/<token>`
  → constructs `File` objects in extension-permission context
- `window.postMessage` ISOLATED → MAIN: `File` objects survive structured-clone with bytes intact
- `content-main.js` builds `DataTransfer`, calls `input.files = dt.files` (spec-compliant
  setter that updates WebKit's internal `[[Files]]` slot — see Phase 7 bug #2 below)
- Dispatches `input` + `change` events
- 200 ms validation probe reads `input.validationMessage` + sibling `[role=alert]` / `[aria-invalid]`

### Phase 0 spike (architectural gate)

Two assumptions were unverified in any prior 5A.* code; both were empirically
validated against the v0.1.23 release-mode build by `test/e2e/5A1-phase0-spike.test.ts`:

1. **Content-script `fetch('http://127.0.0.1:19475/...')`** works under Safari
   Web Extension CSP (manifest's `connect-src` extends to content scripts).
2. **`File` objects survive ISOLATED → MAIN structured-clone** via `window.postMessage`
   with bytes intact (verified by 8-byte `SPFUBYTE` signature comparison in MAIN).

The spike scaffolding (`__SP_FILE_UPLOAD_PROBE_TEST__` sentinel) ships in the
extension bundle as a permanent diagnostic.

### Test coverage

- 5A.1 e2e: **11 PASS, 3 SKIPPED** in `test/e2e/5A1-file-upload.test.ts`
- 5A.1 phase-0 spike: **2 PASS** in `test/e2e/5A1-phase0-spike.test.ts`
- Unit: **392 PASS** (full TS suite)
- Daemon: **153 PASS** (full Swift suite)

The 3 skipped e2e tests document architectural / framework limits, not regressions:

- **`renders RHF label-locator path`** — `label` locator type isn't implemented
  in the extension's inline locator (only `selector` / `xpath` / `ref`). Outer
  `<label>` works by selector; pure label-text traversal does not.
- **`detached-element race`** — element re-resolves at inject time; the race
  window between probe and inject can produce success when the test expects
  detection. Architectural — element identity is not pinned across the
  probe→stage→inject pipeline.
- **`concurrent multi-MB upload`** — NDJSON pipe-write atomicity at daemon
  stdin (PIPE_BUF=4096 on macOS); >4 KB writes from concurrent senders may
  interleave. Single-file writes are atomic; concurrent multi-MB requires
  application-layer framing (deferred).

### Phase 7 fix bundle (real bugs found during verification)

Both fixes shipped in `7f30638`:

1. **`src/tools/extraction.ts` — IIFE wrapper** broke spike sentinel prefix-match.
   `isHarness` bypass only included `__SP_TEST_HARNESS__:`. Extended
   `isSentinelBypass` to also recognise `__SP_FILE_UPLOAD_PROBE_TEST__`. Without
   this, the spike sentinel got wrapped in IIFE and reached MAIN-world JS eval
   as `ReferenceError: Can't find variable: __SP_FILE_UPLOAD_PROBE_TEST__`.

2. **`extension/content-main.js` — internal `[[Files]]` slot update.**
   `Object.defineProperty(input, 'files', { value: dt.files })` shadows the
   prototype getter for JS reads but does NOT update WebKit's internal
   `[[Files]]` slot. `new FormData(form)` reads the internal slot, so submit
   saw an empty `FileList` even though `input.files` JS-read returned the file.
   Fix: `input.files = dt.files` (spec-compliant setter); `defineProperty`
   retained as fallback in a `try/catch`.

Only `FILE_UPLOAD_ELEMENT_DETACHED` is `retryable: true`.

### Notable API divergence from Playwright

- `paths: []` is **rejected** (`FILE_UPLOAD_EMPTY_PATHS`), not silently treated
  as a clear. Use `clear: true` for explicit clearing. This removes the
  agent-constructed-`.filter()`-array foot-gun.
