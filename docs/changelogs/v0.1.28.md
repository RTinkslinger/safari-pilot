# v0.1.28 — Locator system v2

> Headline: Playwright-style locator capabilities reach parity. Chain ops (`filter`/`nth`/`first`/`last`/`and`/`or`/`descendant`), strict-mode action enforcement, multi-element extraction (`safari_query_all`), and tab-scoped custom selector engines (`safari_register_selector` with auto-rehydration on navigation) all ship in one release. Tool count 80 → 83. Two latent injection bugs from a prior sprint discovered and fixed during e2e of the new persistence path.

## T77 — Locator chaining

Every locator-using tool gains a `chain` array on its input schema. Each chain op composes against the previous match:

- `filter` — `hasText` / `hasNotText` / `has` (sub-locator) / `hasNot` (sub-locator)
- `nth` — index into the current match set
- `first` / `last` — convenience aliases
- `and` / `or` — logical combinators across two locator descriptors
- `descendant` — re-root subsequent ops at a child locator

The chain is materialised inside the same in-page `generateLocatorJs` walk as the base locator, so it composes through Shadow DOM and frame-aware routing without a second round-trip. The chain field is documented per-tool — the spec reviewer pinned the inputSchema contract before implementation.

**Test coverage:** `test/e2e/T77-locator-chaining.test.ts` — 10 e2e tests against a live fixture (`/t77-list`) covering each op shape plus integration with refs and Shadow DOM.

## T78 — `safari_query_all` multi-element extraction

New tool. Returns an array of refs (and stamped `data-sp-ref` attributes) for every element matching the locator + chain. Closes the only ◆ partial on Cluster 05 of the parity matrix — multi-element extraction was previously achievable only via `safari_evaluate` with hand-rolled JS.

The implementation reuses the `generateLocatorJs` walker with a `// ── T78 splice point ──` marker that branches to a `querySelectorAll`-equivalent terminal step instead of `querySelector`. Refs are minted with the `sp-` prefix (vs. the existing `e-` prefix for snapshot refs) so subsequent tools accept either via `buildRefSelector`.

**Test coverage:** `test/e2e/T78-query-all.test.ts` — 6 e2e tests against `/t78-grid`.

## T79 — Custom selector engines (`selectorPack`)

Two new tools, gated behind `safari_pilot.config.json → selectorPack.enabled` (default `false`):

- `safari_register_selector` — register a JS function body executed in page context as `new Function("root", "arg", body)`. Returns an Element or null. Body validated for `eval` / `new Function` / `import()` patterns + `MAX_PACK_BODY_BYTES` (32 KB). Name validated against `^[a-zA-Z_][a-zA-Z0-9_]{0,63}$`.
- `safari_unregister_selector` — removes both the storage key and the page-side `__sp_pack[name]` entry.

Pack references work through any locator-using tool: pass `selector: "pack:<name>=<arg>"`. The resolver (`resolveMaybePackSelector` in `src/locator.ts`) calls the pack body in-page, stamps a `data-sp-ref` on the returned element, and rewrites the selector to `[data-sp-ref="..."]` for the actual tool execution.

`safari_register_selector` is on the **HumanApproval** sensitive-action list — the body is a JS injection surface, treated like a privileged write.

### Tab-scoped persistence (Cluster D)

The original spec promised pack persistence across navigations. The first pass shipped only page-scope registration (lost on every reload). Cluster D delivers the spec'd guarantee:

- Registration writes `sp_pack_<tabId>_<name>=body` to `browser.storage.local` AND injects `window.__sp_pack[name] = new Function(...)`. Both happen in one round-trip via the `__SP_PACK_REGISTER__:<json>` sentinel intercepted in `extension/background.js`.
- `tabs.onUpdated` listener: on `status === 'complete'`, reads every `sp_pack_<tabId>_*` key for the tab and re-injects each pack into the new page. Trace events: `pack_rehydrated`, `pack_rehydrate_failed`.
- `tabs.onRemoved` listener clears `sp_pack_<tabId>_*` on tab close. (Shipped earlier in C-8.)

### Latent IIFE bugs found during D-6 e2e

Cluster C's injection script wrapped the page-side write in `(function(){...; return X;})();`. Page-side eval is `new Function(script)()` — the script body IS the function body, so the IIFE expression-statement result is discarded. Both layers — `resolveMaybePackSelector` in `src/locator.ts` AND the three injection scripts in `extension/background.js` (register, unregister, rehydrate) — silently registered nothing on the page side. Unit tests stubbed the engine; the C-9 e2e never reached happy path because HumanApproval blocked the register call.

The fix: change all four injection sites to use top-level `return` (no IIFE wrapper). Caught only when D-6 actually exercised the happy path against real Safari.

**Test coverage:** `test/e2e/T79-selector-pack.test.ts` (3 tests, HumanApproval gate) + `test/e2e/T79D-pack-persistence.test.ts` (4 tests including the navigation-rehydrate guarantee). Source-grep tests in `test/unit/extension/t79d-pack-persistence.test.ts` and `test/unit/tools/selector-pack-sentinel-wire.test.ts` pin the wire format and rule out IIFE regression.

## T80 — Strict-mode action enforcement

Every locator-aware action (click, fill, hover, type, etc.) now throws `StrictnessViolationError` (code `STRICTNESS_VIOLATION`) when the resolved locator matches multiple elements. Previously the engine silently acted on the first match, which is the single most common source of bugs in Selenium-style suites. Playwright has had this since 2022; Safari Pilot now matches.

`safari_query_all` is the explicit opt-out — when you genuinely want all matches.

**Test coverage:** Folded into `test/e2e/T77-locator-chaining.test.ts` (the chaining and strict-mode test paths share a fixture).
