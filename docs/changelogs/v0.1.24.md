# v0.1.24 — Storage-quota recovery + release SOP

> Headline: extensions wedged with "wake-without-reconcile" symptoms self-heal on first wake under v0.1.24. The release pipeline gained a local pre-tag check that mirrors every CI verify step.

## Headline fix: T67 — storage quota in housekeeping must not block reconcile

**The bug (32-hour wedge observed live on 2026-05-03):** when `browser.storage.local` filled to Safari's ~5 MB cap, every `storage.local.set()` call threw "Exceeded storage quota". `wakeSequence` ran `loadTabCache → gcPendingStorage → cleanupStaleStorageBus → connectAndReconcile` in a single outer try/catch. `gcPendingStorage`'s `writePending` threw on quota. The throw aborted the chain BEFORE `connectAndReconcile()` ran. `/connect` never landed on the daemon. `lastReconcileTimestamp` flatlined. `isConnected` stayed `false` for 32+ hours despite alarms firing every minute. Trace evidence at `~/.safari-pilot/daemon-trace.ndjson:109513+`.

This was misnamed at filing as a "wake-without-reconcile" wedge — it isn't. The system was running flawlessly through a code path that didn't include the reconcile step. T60's fix did not apply (different bug class).

**The fix (extension/background.js, 3 surgical changes):**
1. **Reorder `wakeSequence`:** `loadTabCache → connectAndReconcile → gcPendingStorage → cleanupStaleStorageBus`. Reconcile is now critical-path; housekeeping is best-effort.
2. **Per-step try/catch with step-tagged trace events** (`wake_load_error`, `wake_reconcile_error`, `wake_gc_error`, `wake_cleanup_error`). Closes the diagnostic-blindness gap that hid this for 32 hours — operators can now identify which step failed without reading source.
3. **`writePending` quota recovery** mirroring `saveTabCache` (extension/background.js:43-58): catch quota → `remove(STORAGE_KEY_PENDING)` → retry `set` once → swallow on second failure → re-throw non-quota.

**Existing wedged installs auto-recover on the first wake under v0.1.24** — the new emergency-cleanup path empties the bloated pending dict, freeing storage, and the next `connectAndReconcile()` lands. No manual storage clear needed. Verified empirically against a live wedge: `lastReconcileTimestamp` advanced 17 ms after the first v0.1.24 alarm fire.

**Test coverage:** 6 unit tests in `test/unit/extension/t67-storage-quota-blocks-reconcile.test.ts`:
- 4 structural invariants (reorder, per-step traces, quota recovery semantics)
- 1 defense-in-depth invariant (`connectAndReconcile` does NOT call `writePending`)
- 1 behavioral test that extracts `writePending` source via regex, eval-sandboxes it with a stubbed `browser.storage.local`, throws "Exceeded storage quota" on first set call, and asserts (a) no error escapes, (b) `remove()` was called, (c) `set()` was called twice (initial + retry).

`test-reviewer` full-mode review: REVISE round 1 (2 CRITICAL weak oracles), PASS round 2 after strengthening recovery-semantics oracle and adding the behavioral test.

## Release SOP codified

Two CI publish failures during v0.1.24's first attempt — both root-caused to verification gates that had been latent since their addition because no real tag had pushed through CI since their introduction:

1. **`bin/Safari Pilot.zip` contained 45 of 91 entries as AppleDouble (`._*`) metadata.** `scripts/build-extension.sh` used `ditto -c -k --keepParent` without metadata-stripping flags. CI's T47 verify step rejected the bundle as "a sealed resource is missing or invalid". Fixed by adding `--norsrc --noextattr --noqtn --noacl`.
2. **`prepublishOnly` hook (`hooks/pre-publish-verify.sh`) blocked CI's `npm publish`** because it required a `.verified-this-session` marker file that's only created locally by `npm run verify:extension:smoke`. Fixed by short-circuiting on `CI=true` / `GITHUB_ACTIONS=true` (CI's T47 verify is the equivalent gate).

The deeper issue: the last successful release through `release.yml` was v0.1.4 on 2026-04-12. Every tag-driven release between that and v0.1.24 was either skipped, drafted, or never reached the registry. Local rehearsals against the in-place `.app` bypassed both gates. The CI extension-verify and the prepublish hook had never executed against a real tag.

**Fix: `scripts/pre-tag-check.sh`** — 9 local checks mirroring every CI verify step:

1. Working tree clean (excluding gitignored)
2. `package.json` and `extension/manifest.json` versions match
3. `bin/Safari Pilot.app` signed + entitled (sandbox + network.client) + stapled
4. `bin/Safari Pilot.zip` free of AppleDouble (`._*`) metadata
5. Extracted `.zip` survives `codesign --verify --deep --strict`
6. `bin/SafariPilotd` present and executable (universal binary preferred)
7. Unit tests pass
8. Tag for current version not yet created (locally or remote)
9. Prepublish hook short-circuits on `CI=true`

Runs in seconds. Catches the failure modes that wasted CI cycles. Mandatory before any tag push (CLAUDE.md hard rule #9).

**`CLAUDE.md` "Extension Build: Hard Rules"** extended with rules #8–#10:
- #8: NEVER use plain `ditto -c -k --keepParent` — always include `--norsrc --noextattr --noqtn --noacl`.
- #9: NEVER push a release tag without running `bash scripts/pre-tag-check.sh` first.
- #10: CI release workflow must skip the local-only prepublish hook.

A new "Release SOP" subsection in CLAUDE.md documents the step-by-step the next release follows.

## Known limitations carried from v0.1.23

- **T66 (open):** `safari_file_upload` content-isolated.js byte-fetch is rejected on strict-CSP origins (Gmail-class, Trusted Types). Phase 0 spike validated only against the in-repo fixture origin which has permissive CSP. Filed as a P2 follow-up; docs/TRACKER.md has the full triage notes. Workaround: tell the user to drag the file in for these origins.
- **3 documented `safari_file_upload` test skips:** RHF label-locator (extension JS only supports `selector`/`xpath`/`ref`), detached-element race (re-resolves at inject time), concurrent multi-MB pipe atomicity (NDJSON pipe-write at daemon stdin, PIPE_BUF=4096 on macOS). All in `test/e2e/5A1-file-upload.test.ts`, all marked `it.skip` with documented reasons.
