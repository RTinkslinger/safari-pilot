## Headline fix 1: T72 — pollLoop overlapping fetches

**The bug:** when running 4+ e2e test files in a single vitest sweep, ~80% of runs surfaced a single test failure that migrated between tests run-to-run (5A.3 in one sweep, 5A.9 in another, initialization in another). Phase-1 systematic-debugging traced the cause to MV3 event-page suspension between sequential `httpPoll` iterations: in the microtask gap between fetch.then() resolving and the next fetch.start(), Safari can suspend the event page. A command queued in the daemon bridge during the suspension waited until the next chrome.alarms tick (~10s) woke the page. Daemon-trace evidence at `~/.safari-pilot/daemon-trace.ndjson` shows status_polled gaps of exactly 10s correlating with command-stuck windows.

**The fix (extension/background.js, pollLoop):**
- Pre-launch the FIRST `httpPoll` BEFORE entering the while loop: `let inflightPoll = httpPoll(abortSignal)`.
- Re-launch the next fetch INSIDE the loop body BEFORE processing this batch's commands: `inflightPoll = httpPoll(abortSignal)` runs before any `await executeCommand(cmd)`.
- Re-arm in `TimeoutError` and `BACKOFF_MS` catch branches so the next iter awaits a fresh fetch.

A fetch is now always in flight. The MV3 event page never sees an idle moment. Validation: 5x sweep 3 PASS / 2 FAIL = 40% (was 80%).

**Test coverage:** `test/unit/extension/t72-pollloop-overlapping-fetches.test.ts` — 7 source-grep tests guarding the structural invariants (matches T60 precedent style). Two-pass test-reviewer gate: REVISE round 1 (CRITICAL "let-binding could be a decoy" + MAJOR "sibling-try evasion"), PASS round 2 after tightening test 7 to require `await inflightPoll` is consumed AND inside the BACKOFF_MS try block.

## Headline fix 2: T73 — supersedePollLoop skips abort when pollLoop is healthy

**The residual bug after T72:** flake rate dropped 80% → 40% with overlapping fetches, but didn't go to zero. Daemon-trace showed: `alarm_fire` at minute boundaries → `pollloop_aborted reason=abort_signal` → 10s gap with no extension activity → command queued in bridge with no waiter → `cmd_dispatched` only after a subsequent execute. The keepalive alarm (KEEPALIVE_PERIOD_MIN=1) was unconditionally calling `supersedePollLoop` every 60s — killing the in-flight `/poll` fetch even when pollLoop was healthy.

**The fix (extension/background.js):**
- Module-level `let lastSuccessfulPoll = 0` updated inside pollLoop's success try-block AFTER the inflight-await resolves.
- `supersedePollLoop` reads it and skips the abort cascade when fresh: `if (pollLoopController && lastSuccessfulPoll > 0 && Date.now() - lastSuccessfulPoll < 30_000) { emitTrace('pollloop_supersede_skipped', ...); return; }`.
- T60 wedge-recovery contract preserved on the unhealthy path (abort + new AbortController + new pollLoop still fire when the predicate fails).

30s threshold rationale: daemon long-poll holds 5s normally; healthy pollLoop resolves on every command burst AND every 5s during idle. 30s gives 6x headroom while still catching genuinely-wedged loops within one keepalive cycle on the second tick.

**Validation: 10x e2e sweeps PASS (10/10), flake rate 0%.** Full journey: 80% (pre-T72) → 40% (T72-partial) → **0% (T73)**.

**Test coverage:** `test/unit/extension/t73-supersede-skip-when-healthy.test.ts` — 6 source-grep tests. One-pass test-reviewer: PASS with two MAJOR tightening recommendations (test 2 "set in catch" evasion + test 6 "any literal" permissiveness), both addressed before GREEN.
